Shop Inventory Management System — System Design
1. Objective

Design a system where a shop owner (admin) can manage multiple branches, each branch having a manager and multiple salesmen.

The system will support:

inventory tracking per branch

sales approval workflow

stock order workflow

analytics at branch and shop level

notifications (later)

future SaaS support (multiple shops)

2. User Roles

The system contains three roles.

Admin

Represents the shop owner.

Capabilities:

manage branches

create managers and salesmen

approve or reject stock requests from branches

view analytics of all branches

manage product catalog

Manager

Responsible for one branch.

Capabilities:

approve or reject sales submitted by salesmen

negotiate sales requests

view branch inventory

request stock from admin

view branch analytics

Salesman

Works under a branch manager.

Capabilities:

create sales requests

negotiate with manager

track their sales

3. Hierarchy Structure

The system hierarchy:

Shop
 ├── Branch 1
 │     ├── Manager
 │     ├── Salesman
 │     └── Inventory
 │
 ├── Branch 2
 │     ├── Manager
 │     ├── Salesman
 │     └── Inventory

Future SaaS architecture:

Platform
 ├── Shop A
 │     ├── Branches
 │
 ├── Shop B
 │     ├── Branches

Each shop's data is isolated using shop_id.

5. Workflow Design

The system contains two separate workflows.

1️⃣ Sales Workflow
2️⃣ Order Request Workflow

Each workflow is implemented using a state machine model.

Transitions are stored in a transition history table.

10. Inventory Updates

Inventory changes happen only when records are finalized.

Sales Approved
inventory.quantity -= sold_quantity
Order Approved
inventory.quantity += ordered_quantity

These operations must run inside database transactions.

14. Key Design Decisions

The system is designed around:

workflow-driven architecture

state machines

request → record pattern

branch-level inventory management

multi-product sales transactions

SaaS scalability

This approach avoids simple CRUD limitations and models real business processes.
